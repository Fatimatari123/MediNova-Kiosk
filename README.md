# MediNova - AI-Powered Health Kiosk

## Project Overview

MediNova is an AI-powered desktop health kiosk designed to provide basic medical guidance to rural and underserved communities in Pakistan. The system allows patients to describe their symptoms through voice input in Urdu, receives an AI-assisted triage assessment, and generates a  health report text file of the booked appointment at the send of the session. The goal is to reduce the gap between patients and healthcare guidance in areas where clinics, hospitals, and literate assistance are not easily accessible.

The application runs entirely on a desktop machine and is designed to be simple enough for users with no prior experience with technology. It requires no typing and no reading ability, as all interaction is done through voice.

## Problem Statement

Millions of people living in rural and underserved areas of Pakistan lack access to even basic healthcare guidance. Factors such as low literacy rates, language barriers, distance from medical facilities, and high consultation costs prevent people from seeking timely help. MediNova addresses this by placing an intelligent, voice-driven health assistant on any standard computer, enabling patients to describe how they feel in Urdu and receive structured guidance without needing a doctor present. MedinNova itself cannot replace a doctors recommendation.


## How the Application Works

When the application is launched, a graphical window opens with a microphone button at the center. The patient clicks the button, and the kiosk greets them in Urdu and asks how they are feeling. Everything from this point onward is handled through voice.

The system listens to the patient using the device microphone, sends the audio to Google Speech Recognition (configured for Urdu), and converts it to text. The text is then passed through a logic engine that determines what stage of the conversation the patient is at and what response to generate.

The conversation follows a structured flow. First, the system asks about the general complaint. If the patient says they are unwell, it identifies which part of the body is affected. It then asks a series of follow-up questions such as when the pain started, whether it is constant or comes and goes, what the severity is, and whether there are any additional symptoms. Based on the answers collected, the system matches the described symptoms against a custom disease database and assigns a triage level.

All responses from the kiosk are spoken aloud in Urdu using Google Text-to-Speech. The patient never needs to read anything on the screen.

At the end of the session, if the patient agrees to book a doctor appointment, the system collects their name and age and generates a text file of the health report containing all the information gathered during the session, including the recommended specialist and urgency level.


## Triage System

The triage system works on three levels:

GREEN means the condition appears manageable at home. The system provides home care advice and ends the session.

YELLOW means the condition requires monitoring and a doctor visit is recommended within a reasonable timeframe. The system suggests booking an appointment.

RED means the condition is serious and requires immediate medical attention. The system strongly urges the patient to see a doctor or go to an emergency facility right away.

The urgency level is determined by two things working together: the disease rule matched from the custom database, and the pain severity reported by the patient themselves. If the patient reports severe pain but the matched disease rule says low urgency, the system takes the more critical of the two levels as the final result.


## Custom Database and Hardcoded Logic

A significant portion of the application logic is hardcoded and rule-based rather than dynamically generated by AI. The system uses two text files as its knowledge base.

The first file, disease_rules.txt, contains a list of diseases along with their associated symptoms in Urdu, their urgency level, and either a recommended specialist or home care advice. Each entry follows a pipe-separated format. For example, a rule for Heart Attack contains the symptoms of severe chest pain, difficulty breathing, and sweating, marks it as Critical, and recommends an immediate visit to a Cardiologist. The system reads this file at startup and stores all rules in memory.

The second file, dht_rules.txt, contains a home care knowledge base covering common conditions such as headache, fever, stomach pain, cold and flu, and skin rash. Each entry provides a brief explanation of the condition in Urdu, recommended home care steps, and a warning sign that should prompt the patient to see a doctor.

When a patient describes their symptoms, the application tokenizes both the patient input and the disease rule symptoms, then calculates a match score. If the match score is 60 percent or higher, the system considers it a confident match and presents its analysis immediately. If the score is lower, it generates targeted follow-up questions based on the missing symptoms from the closest matching rule, collects those answers, and then finalizes the triage.

The follow-up questions, the Urdu phrases used throughout the conversation, the body part keywords, and the urgency mapping are all hardcoded within the application code. The AI component (Groq API) is integrated but the core triage and conversation flow does not depend on it for the rule-based logic path.


## Limitations

The application currently has several known limitations that the team acknowledges.

The Urdu text-to-speech uses Google's free gTTS service, which produces an automated accent that does not always sound natural. There are occasional pronunciation issues, gender inconsistencies in spoken Urdu, and some repeated phrasing in the conversation flow. These are limitations of the free TTS engine used and not of the conversation logic itself.

The disease database is small and manually curated. It currently covers six diseases in the main rules file and five conditions in the home care knowledge base. This means the system may not recognize symptoms for conditions not included in these files.

The speech recognition depends on Google's online API, which requires an active internet connection. In areas with poor connectivity, voice input may fail or produce incorrect transcriptions.

Body part detection and symptom tokenization are based on a fixed list of Urdu keywords. Dialects, informal speech, or mixed Urdu-English input may not always be recognized correctly.


## Future Improvements

With more time and resources, the team would like to explore the following improvements:

Replacing gTTS with a higher quality Urdu TTS engine that produces more natural and gender-appropriate speech.

Expanding the disease rules database to cover a broader range of conditions, ideally sourced from verified medical references.

Adding offline speech recognition so the kiosk can function without an internet connection, which is critical for rural deployment.

Integrating a real appointment booking backend connected to local healthcare providers.

Adding support for regional languages spoken in Pakistan beyond Urdu.

Developing a physical kiosk prototype with a touchscreen interface, built-in microphone, and thermal printer for the health report.

## Tech Stack and Tools

Python 3.12.7.

The graphical interface is built using Python's built-in Tkinter library.

Voice input is handled by the SpeechRecognition library combined with SoundDevice for microphone access, using Google's Speech Recognition API configured for Urdu (ur-PK).

Voice output is handled by gTTS (Google Text-to-Speech) and played back using Pygame's audio mixer, with Pydub used for audio processing and speed adjustment.

The Groq API provides access to the LLaMA language model for AI-assisted responses.

FFmpeg is bundled using the static-ffmpeg package to handle audio format conversion.

Environment variables and API keys are managed using python-dotenv.

To set up the project, the user needs to install Python, clone the repository, create a virtual environment, install the dependencies from requirements.txt, copy the .env.example file to .env, and add their Groq API key. A free Groq API key can be obtained from console.groq.com. After that, the application is launched by running main.py.


## Setup Instructions

Install Python 3.12.7.

Clone the repository:
git clone https://github.com/Fatimatari123/MediNova-Kiosk.git
cd MediNova-Kiosk

Create and activate a virtual environment:
python -m venv .venv
.venv\Scripts\activate        (Windows)
source .venv/bin/activate     (Mac/Linux)

Install dependencies:
pip install -r requirements.txt

Set up your API key:
Copy .env.example to .env and fill in your Groq API key.

Run the application:
python main.py
